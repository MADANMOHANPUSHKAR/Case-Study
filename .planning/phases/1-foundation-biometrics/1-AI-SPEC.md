## 1. System Classification

**System Type:** In-Browser Client-Side ML (Facial Recognition & Liveness)

**Description:**
A client-side biometric system that uses the user's web camera to extract mathematical facial features (embeddings) and detect liveness (e.g., blinking). It runs entirely in the browser using WebAssembly to protect user privacy and save server costs. 

**Critical Failure Modes:**
1. **False Acceptance (Spoofing):** Accepting a photograph or video of a user as a live scan.
2. **False Rejection:** Consistently failing to recognize a legitimate user due to poor lighting or camera quality, blocking them from accessing campus facilities.
3. **Privacy Leak:** Accidentally transmitting or logging raw webcam images (JPEGs/PNGs) instead of mathematical embeddings.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Higher Education / Campus Operations

**User Population:** College students accessing gyms, swimming pools, and society events via their mobile phones or laptops.

**Stakes Level:** Medium (prevents minor fraud/proxy attendance; not a life-or-death security system).

**Output Consequence:** A student is granted or denied entry to a facility/event.

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|-----------------------|--------------------|--------|--------|
| Anti-Spoofing | Rejects high-quality printed photos and iPad videos | Accepts a photo held up to the camera | High | Admin Requirement |
| Latency | Scan completes in < 2 seconds | Scan takes > 5 seconds, causing queues | Medium | UX Requirement |
| Lighting Tolerance | Works in dim gym lobbies | Fails unless user is in direct sunlight | High | Operational Reality |

### Known Failure Modes in This Domain

- **The "Friend Proxy":** Students handing their phone to a friend to check them in. (Mitigated by matching the scanned face to the logged-in user profile's embedding).
- **The "Photo Attack":** Holding up a student ID card photo to the webcam.

### Regulatory / Compliance Context

- **FERPA / GDPR / Local Privacy Laws:** Biometric data requires explicit opt-in consent. We mitigate risks by storing **only** mathematical embeddings in `pgvector`, never raw images.

### Domain Expert Roles for Evaluation

| Role | Responsibility |
|------|---------------|
| Campus IT Security | Validating the anti-spoofing constraints and network privacy. |
| Student Volunteers | Beta testing the scanning flow in various lighting conditions. |

## 2. Framework Decision

**Selected Framework:** `@mediapipe/tasks-vision` (Google MediaPipe)

**Version:** `0.10.x`

**Rationale:**
MediaPipe provides robust, production-ready WebAssembly (WASM) models for Face Detection and Face Landmarking directly in the browser. It is faster and more actively maintained than older libraries like `face-api.js`, providing excellent mobile performance.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| `face-api.js` | Unmaintained, heavy, and significantly slower on mobile devices compared to MediaPipe. |
| AWS Rekognition | Requires sending video frames/images to the cloud. High cost, high latency, and massive privacy concerns. |

**Vendor Lock-In Accepted:** Partial — We rely on Google's pre-trained MediaPipe models, but they are open-source and run locally.

## 3. Framework Quick Reference

> Fetched from official docs by `gsd-ai-researcher`. Distilled for this specific use case.

### Installation
```bash
npm install @mediapipe/tasks-vision
```

### Core Imports
```javascript
import { FaceLandmarker, FilesetResolver } from "@mediapipe/tasks-vision";
```

### Entry Point Pattern
```javascript
const vision = await FilesetResolver.forVisionTasks(
  "https://cdn.jsdelivr.net/npm/@mediapipe/tasks-vision@0.10.x/wasm"
);
const faceLandmarker = await FaceLandmarker.createFromOptions(vision, {
  baseOptions: {
    modelAssetPath: `https://storage.googleapis.com/mediapipe-models/face_landmarker/face_landmarker/float16/1/face_landmarker.task`,
    delegate: "GPU"
  },
  runningMode: "VIDEO",
  numFaces: 1
});

// Inside video loop
const results = faceLandmarker.detectForVideo(videoElement, performance.now());
```

### Key Abstractions
| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `FilesetResolver` | Loads the WASM binaries. | During initialization. |
| `runningMode: "VIDEO"` | Optimizes tracking across consecutive frames. | When reading from a webcam stream. |

### Common Pitfalls
1. Failing to handle browser permissions gracefully (`NotAllowedError`).
2. Not waiting for the `loadeddata` event on the HTML `<video>` element before processing.
3. Memory leaks from not cleaning up the WASM instances or stopping the camera tracks when the component unmounts.

## 4. Implementation Guidance

**Model Configuration:**
- `numFaces: 1` — Reject frames with multiple faces to prevent confusion.
- `minFaceDetectionConfidence: 0.7` — Prevent false positives from background noise.

**Core Pattern:**
Use React `useEffect` to request camera permissions, attach the stream to a hidden `<video>` element, and use `requestAnimationFrame` to continuously feed frames to MediaPipe.

**State Management:**
Keep the camera stream state local to the `Scanner` component. Once liveness is verified and the embedding is extracted, send the embedding to the Supabase Edge Function via a POST request.

**Context Window Strategy:**
N/A (Not an LLM).

## 5. Evaluation Strategy

### Dimensions

| Dimension | Rubric (Pass/Fail or 1-5) | Measurement Approach | Priority |
|-----------|--------------------------|---------------------|----------|
| False Acceptance Rate (FAR) | Fail if a photo tricks the system. | Human testing with printed photos / iPads. | Critical |
| False Rejection Rate (FRR) | Fail if < 90% of legitimate scans pass. | Beta testing logs. | High |
| Latency | Pass if Time-To-Verify < 3s. | Telemetry on scanner component. | Medium |

### Eval Tooling

**Primary Tool:** Custom telemetry logged to Supabase + Manual Human Testing Matrix.

**Setup:**
A dedicated testing table in Supabase tracking `scan_attempts`, `lighting_condition`, and `success`.

### Reference Dataset

**Size:** N/A for continuous testing, but initial testing requires 10 distinct users testing 5 scenarios each.

**Composition:**
- Direct lighting
- Backlit (window behind user)
- Wearing glasses / no glasses
- Photo spoof attempt

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| Multi-Face Block | `results.faceLandmarks.length > 1` | Block scan, prompt user to ensure only they are in frame. |
| Time-out | Scan fails to detect liveness after 10s | Escalate to OTP email fallback. |

## 7. Production Monitoring

**Tracing Tool:** Supabase Logs / Vercel Analytics

**Key Metrics to Track:**
- Ratio of successful face logins vs. OTP fallbacks.
- Average time spent on the scanner page.
- Error rates for camera permissions.

**Alert Thresholds:**
- Alert if OTP fallback rate exceeds 30% globally (indicates a breaking change in the AI model or a widespread browser issue).
